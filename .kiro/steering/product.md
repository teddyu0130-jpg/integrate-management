# プロダクト概要

App③（integrate-management）は「家ものメイドくん」（ODS-RAM 家庭スケール POC）の **Federation Engine** です。永続データを自身では持たず、独立した Provider アプリ（App①・App②）から IRI キーで横断的にデータを結合し、意味の辞書（YAML）に従って統合ビューを生成するステートレスな統合・示唆アプリです。8 画面（I-01〜I-08）を提供しますが、書き込みはすべて辞書のマッピングに従い各 Provider へ委譲します。

## 主要ケーパビリティ

- **IRI による横断 Federation**：リクエスト時に複数 Provider の REST エンドポイントを並列取得し、共有 IRI キーでレコードを突き合わせて統合ビューを生成（`name` / `category` / `location` / `stock_level` / `note` / `updated_at`）
- **辞書駆動の読み書き**：属性マッピングと書き込み先（`writable_to`）を YAML で宣言。Provider 追加・書き込み先変更はコード変更なしに辞書追記だけで対応
- **委譲書き込み**：残量変更・登録・編集・削除は App③ が直接書かず、辞書の `writable_to` マッピングに従い対象 Provider の Data Endpoint を呼ぶ（location 系 → App①、stock_level 系 → App②）
- **2 視点の射影**：同一の統合データセットを、夫（場所グループの「探すモード」）と妻（残量順の「補充モード」）の異なるビューとして描画
- **横断的示唆**：buy 候補リスト（残量 `low` / `empty` + 戻し場所を一覧化）、重複登録の検出（`name` 一致 & `iri` 不一致）——単独 Provider では生成不能
- **障害許容の縮退表示**：Provider が応答しない場合、取得できたデータだけで表示を継続し、欠落属性を明示

## 対象ユースケース

- アプリをまたいだ在庫の場所・残量の確認（「シャンプーはどこにある？」「残量は？」）
- 残量 `low` / `empty` のアイテムを収納場所付きで一覧化した買い物リスト生成（FR-05）
- Provider 単独では不可能な重複登録の検出——統合層でのみ成立（FR-11）
- 登録・編集・削除・残量変更を 1 アプリから行い、変更が各 Provider へ委譲書き込みで反映（FR-10）
- ※ 消費ペース予測（`days_remaining`）は Step 2 以降のロードマップ。MVP スコープ外

## 価値提案

App③ は ODS-RAM の核心仮説を実証します。**「Provider が増えるほど、エンジンコードを一切変えずに App③ が賢くなる」** ——新しい価値は辞書への追記だけで注入されます。「データは生まれた場所に置いたまま、意味だけを旅させる」原則の家庭スケール実証です。
